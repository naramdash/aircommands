# 01. 제품 요구사항

## 1. 문제 정의

사용자는 키보드나 마우스 없이 손가락 움직임만으로 로컬 애플리케이션을 실행하고 싶다. 현재 프로젝트는 웹캠 기반 손 추적과 서버 측 앱 실행 기능을 각각 일부 가지고 있지만, 둘을 안정적으로 연결하는 명령 인식 계층이 없다.

## 2. 제품 목표

오른손 검지 제스처를 이용해 명령 후보를 만들고, 사용자가 확인 동작을 수행하면 서버에 요청을 보내 등록된 로컬 앱을 실행한다.

## 3. 핵심 사용자

- 개발 중인 사용자가 로컬 컴퓨터에서 제스처 명령 UX를 검증한다.
- 카메라 앞에서 손을 움직일 수 있고, 명령 실행 결과를 화면에서 확인할 수 있다.
- 잘못된 앱 실행을 막기 위해 후보 명령 표시와 확인 동작을 받아들일 수 있다.

## 4. MVP 사용자 여정

1. 사용자가 웹앱을 연다.
2. 브라우저가 카메라 권한을 요청한다.
3. 사용자가 오른손 검지를 화면 안에 둔다.
4. 화면에 손 추적 상태와 최근 검지 경로가 표시된다.
5. 사용자가 검지를 위/아래/왼쪽/오른쪽으로 스와이프한다.
6. 클라이언트가 스와이프를 후보 명령으로 표시한다.
7. 사용자가 검지를 0.8초 이상 멈춰 확인한다.
8. 클라이언트가 서버 API로 앱 실행 요청을 보낸다.
9. 서버가 allowlist에서 앱 실행 명령을 찾아 실행한다.
10. 클라이언트가 성공, 실패, cooldown 상태를 표시한다.

## 5. MVP 기능 요구사항

### FR-1. 카메라와 손 추적

- 브라우저 카메라 권한을 요청한다.
- 오른손을 우선 명령 입력 대상으로 사용한다.
- MediaPipe `HandLandmarker`의 `detectForVideo` 결과에서 오른손 landmark를 읽는다.
- 오른손 검지 끝 landmark index 8번을 매 프레임 기록한다.
- 최근 5초 경로는 화면에 trail로 표시한다.

### FR-2. 제스처 후보 인식

- 최근 0.8초의 검지 경로를 사용해 스와이프 후보를 계산한다.
- 지원 방향은 `swipe_up`, `swipe_down`, `swipe_left`, `swipe_right`이다.
- 너무 짧은 이동, 느린 이동, 대각선 흔들림은 후보로 인정하지 않는다.
- 후보는 최소 2프레임 이상 같은 방향으로 유지되어야 한다.

### FR-3. 명령 매핑

- 제스처와 앱 실행 명령을 고정 매핑한다.
- MVP 기본 매핑:
  - `swipe_up` -> `chrome`
  - `swipe_down` -> `notepad`
  - `swipe_right` -> `vscode`
  - `swipe_left` -> `terminal`
- 매핑은 클라이언트 상수로 시작하고, 추후 설정 파일로 분리할 수 있어야 한다.

### FR-4. 실행 확인

- 후보 명령만으로는 서버 요청을 보내지 않는다.
- 후보 명령이 표시된 뒤 검지 끝이 0.8초 이상 작은 반경 안에 머무르면 실행을 확정한다.
- 후보 발생 후 1.2초 안에 확인되지 않으면 후보를 취소한다.
- 손이 화면에서 사라지면 후보를 취소한다.

### FR-5. 서버 통신

- 클라이언트는 앱 실행 확정 시 `POST /api/apps/open`으로 요청한다.
- 요청에는 `app`, `source`, `gesture`, `clientRequestId`가 포함된다.
- 서버 응답의 성공/실패를 화면에 표시한다.
- 기존 `/api/open-chrome`은 호환 API로 유지하거나 새 API에 위임한다.

### FR-6. 중복 실행 방지

- 클라이언트는 앱 실행 후 2초 동안 cooldown 상태에 들어간다.
- cooldown 중에는 새 후보를 만들 수 있지만 실행 요청은 보내지 않는다.
- 서버는 같은 `clientRequestId`가 짧은 시간 안에 반복되면 중복 요청으로 처리한다.

### FR-7. 오류 표시

- 카메라 권한 거부
- 오른손 미감지
- 서버 API 실패
- 미등록 앱 요청
- 앱 실행 command 실패

위 오류는 화면에 한국어로 표시한다.

## 6. 비기능 요구사항

### NFR-1. 안전성

- 서버는 allowlist에 등록된 앱만 실행한다.
- 사용자 입력을 shell command 문자열에 직접 결합하지 않는다.
- 임의 command 실행 기능은 만들지 않는다.

### NFR-2. 응답성

- 제스처 후보는 사용자가 스와이프를 마친 뒤 300ms 이내에 표시되는 것을 목표로 한다.
- 서버 호출 상태는 즉시 `실행 중`으로 표시한다.
- 앱 실행 결과는 API 응답 즉시 표시한다.

### NFR-3. 관찰 가능성

- 화면에 현재 상태, 후보 명령, 확인 진행률, cooldown 여부를 표시한다.
- 개발 중에는 console log 또는 화면 debug panel로 raw 제스처 후보와 confidence를 확인할 수 있어야 한다.

### NFR-4. 테스트 가능성

- 제스처 판정은 DOM/MediaPipe 의존이 없는 순수 함수로 분리한다.
- 서버 API는 실제 앱 실행 없이도 테스트할 수 있도록 command runner를 분리한다.

## 7. 제외 범위

- 사용자별 제스처 학습
- 원 그리기 제스처의 MVP 1차 구현
- 양손 조합 명령
- 음성/키보드 혼합 명령
- 배포 서버에서 외부 사용자가 앱을 실행하는 기능
- 관리자 인증/계정 시스템

## 8. 완료 정의

MVP 완료는 다음을 모두 만족해야 한다.

- 오른손 검지 스와이프 4방향이 후보 명령으로 인식된다.
- 후보 명령과 실행 예정 앱이 화면에 표시된다.
- 확인 동작 전에는 앱 실행 API가 호출되지 않는다.
- 확인 동작 후 서버 API가 호출되고 등록 앱이 실행된다.
- 같은 제스처가 cooldown 중 중복 실행되지 않는다.
- 미등록 앱과 실행 실패가 오류로 표시된다.
- `npm run build`가 성공한다.
- 제스처 판정 순수 함수 테스트가 통과한다.
- 실제 카메라 수동 테스트 체크리스트를 통과한다.
